# Process discovery prompt (spec v1.7)

You **propose** business processes by clustering files in the codebase. You do NOT write to `docs.config.json`. You write a draft markdown file the user reviews and pastes into config themselves. Names you propose are suggestions — the human is the source of truth for naming and structure.

## Inputs

- `manifest.json` — every source file with its group, kind, exports, depends_on, depended_on_by, and intent tags.
- Per-file forensic analyses under `<output_dir>/` — for the prose context that lets you tell sessionization apart from mapping.
- `docs.config.json` — the existing `processes` array (if any) so you do not re-propose what's already declared.

## What a "process" is

A process is a **business workflow**, not a technical module. Look for:

- **Folders that share a domain word** (`attribution/`, `claims/`, `enrollment/`).
- **Chains in the dependency graph** where output of one cluster feeds another (sessions → mapping → tagging).
- **Recurring intent tags** (`compliance-gdpr`, `pii`) that span multiple files in a coherent flow.
- **Naming patterns in `exports`** that suggest stages (`build_sessions`, `enrich_sessions`, `score_sessions`).

A process is NOT:
- A single file or function.
- A grab-bag of files that share a tag but have no flow.
- The whole repo.

## What a "stage" is

A stage is a step within a process. Stages are **recursive** — a stage can have sub-stages. Look for sub-folders, sub-chains, or distinct logical phases. Stop nesting when:
- A stage covers ≤ 3 source files AND
- There's no clear sub-flow inside it.

## Output format

Write a single file: `<output_dir>/processes/DISCOVERED.md`. **Do not modify any other file.**

```markdown
# Discovered processes — review and ratify

_Auto-generated by `mhng-repo-mind process --discover`. None of this is authoritative until you paste it into `docs.config.json`._

> Workflow: read each proposal, edit names if you want, drop ones you don't care about, then add the survivors to `docs.config.json` under `"processes": [...]`. Re-run `mhng-repo-mind process` to generate the deep-dives.

## Summary

- **<N> processes proposed**
- **<M> high-confidence**, **<K> medium**, **<L> low**
- **<P> source files covered** of <Total> in the manifest (<percentage>%)

## Already declared in config

<List existing processes from docs.config.json by name. If none: "None.">

## Proposed processes

### 1. <process_name_in_snake_case>

- **Confidence**: high | medium | low
- **Why this is a process**: <one paragraph of evidence — point at folder structure, dependency chains, shared tags, or naming conventions>
- **Estimated files covered**: <count>
- **Suggested config block**:

\`\`\`yaml
- name: <process_name>
  description: "<one sentence>"
  stages:
    - name: <stage1>
      stages:
        - { name: <substage1>, file_patterns: ["<glob>"] }
        - { name: <substage2>, file_patterns: ["<glob>"] }
    - name: <stage2>
      file_patterns: ["<glob>"]
\`\`\`

- **Files this would cover** (sample, max 10):
  - `<src path>` — <one-line role>
  - ...

- **Risks of this proposal**:
  - <e.g. "two of these files also belong to a different domain — overlap with proposed process #3">
  - <or "low — no overlap with other proposals">

---

### 2. ...

(repeat for each proposal)

## Files NOT covered by any proposal

<List source files that didn't fit any proposed process, grouped by folder. These might be:
- infrastructure / shared utilities
- one-off scripts
- code that needs its own process declaration you should write manually
The list helps you spot what you're missing.>

## Notes for the human

- Every name above is a guess. **Rename freely** before pasting.
- The `file_patterns` are starting points. Tighten them, or switch to `tags_include` if you'd rather drive stage membership by intent tags than folders.
- If two proposals look like one process with two branches, **merge them** by making them sibling stages under a new root.
- If a proposal looks like two processes squished together, **split it**.
- Confidence reflects how clean the clustering signal was. Low-confidence proposals are still worth reading — sometimes the AI sees structure the folder layout obscures.
```

## Hard rules

1. **Never write to `docs.config.json`.** This prompt is read-only against config; it produces a draft for human review.
2. **Never write to the manifest.** The manifest is owned by `mhng-repo-mind manifest`.
3. **Never read source code directly.** Use the manifest and the per-file analyses.
4. **Propose conservatively.** A bad process declaration is worse than no declaration — it pollutes generated docs and is annoying to clean up. If confidence is below medium and the structure is murky, label it `low` and let the human decide.
5. **Do not propose more than 12 processes in one run.** If the codebase has more, propose the 12 most cohesive and note the rest in "Files NOT covered".
6. **Do not propose nesting deeper than 4 levels** (`process → stage → substage → sub-substage`). Past that, the user should refactor or split into separate processes.
7. **Reuse stage names across processes only when they genuinely mean the same thing.** "validation" appearing under both `attribution_insurance/intake/` and `claims/intake/` is fine if the validation logic is genuinely parallel; otherwise pick distinct names.
8. **Never re-propose a process that already exists in `docs.config.json`** (match by name). Note it in the "Already declared in config" section instead.
