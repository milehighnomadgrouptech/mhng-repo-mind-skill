# Project overview prompt (spec v1.2)

Produce **`OVERVIEW.md`** — the single document that answers "what kind of project is this and what does it do?" for someone (or some AI) opening the docs for the first time.

## Inputs

The orchestrator hands you:

- `manifest.json` — full dependency graph
- `INDEX.md` — topic router
- Every `_SUMMARY.md` — per-group rollups
- A list of `top_level_configs`: paths to root-level config files (package.json, tsconfig.json, next.config.*, vercel.json, dbt_project.yml, pyproject.toml, Cargo.toml, go.mod, docker-compose.yml, Dockerfile, etc.)
- A list of `entry_point_analyses`: the per-file analyses for the top ~10 nodes with empty `depended_on_by` (the real surface of the project)

**You read these. You do NOT read every per-file analysis.** The whole point of the layered design is that this synthesis works from summaries + entry points + configs.

Write to `<output_dir>/OVERVIEW.md`:

## Rules

1. **Never invent.** Every claim in the stack table, the architecture paragraph, and the surface area must trace to a config file, an analysis, or the manifest. If you cannot trace it, write "not visible".
2. **Pick the most specific accurate label**, not the most general. "Web app" is too vague; "Next.js 14 App Router with React Server Components, deployed on Vercel" is right.
3. **Configs over summaries.** The top-level configs (`package.json`, `dbt_project.yml`, etc.) are the highest-signal source for the stack table. Only fall back to summaries when configs are absent or ambiguous.
4. **Do not read every per-file analysis.** This document is a synthesis from the layered output below. If you find yourself wanting to read 50 files, your scope is wrong — read the summaries instead.
5. **Length cap**: 200 lines. This is a router, not a textbook.
6. **One file**: write only `OVERVIEW.md`. Do not create or modify other files.
7. **Be honest about what you cannot tell.** The "Open questions" section is mandatory and should not be empty.
