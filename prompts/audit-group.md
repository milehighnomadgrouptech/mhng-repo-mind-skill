# Group audit prompt (spec v1.1)

You receive **every analysis `.md` in one logical group** (a package, app, or layer) plus the group's `_SUMMARY.md`. Your job is to **find bugs**, not summarize.

## Constraints — read carefully

1. **Bugs only.** Code that *will* misbehave on some input, not code that "could be improved", is "not idiomatic", or has "missing types". Style, naming, formatting, and lint-class issues are out of scope — they are caught by other tools.

2. **Every finding must cite at least one analysis file by path.** If you cannot cite, you cannot report.

3. **Every finding must include, in this exact shape:**
   - **title** — one line, imperative voice ("caller fails to check null return of getUser()")
   - **severity** — `critical` | `high` | `medium` | `low`
   - **location** — `source_file:symbol` or `source_file:Lx-y`
   - **blast_radius** — which downstream files are affected (list)
   - **evidence** — 1–3 sentences. Cite the relevant `assumes` / `enforces` / `risks` / `top_suspicion` from the analyses that led you here.
   - **reproduction** — the smallest concrete input, call sequence, or environment that would trigger it. If you cannot name one, write `"not visible — requires runtime trace"`.

4. **Prefer cross-file findings.** The single-file analyzer already looked at each file in isolation. Your unique value is in **matching `assumes` from one file against `enforces` or missing `enforces` in another**. A contract mismatch across files is worth 10 single-file nitpicks.

5. **No speculation.** If you cannot point to a specific symbol, line, or contract mismatch, do not report it.

6. **No duplication of deterministic tools.** Do not report:
   - Type errors a type checker would find
   - Unused variables, unused imports
   - Missing `await`
   - Formatting
   - `console.log` left in code
   If the user's linter/typechecker would catch it, skip it.

7. **False confidence is worse than silence.** If nothing in this group looks concretely wrong, say so and list "what I checked but did not find" as evidence of coverage.

## Output format

Write a single markdown file to the path given as `OUTPUT_PATH` (typically `<group>/_AUDIT.md`):

```markdown
# <group-name> — Audit
_Generated by `mhng-repo-mind audit` on <ISO timestamp>._

## Summary
<1–3 sentences: how many findings, the highest severity, the dominant theme>

## Findings

### 1. <title>
- **severity**: <level>
- **location**: <path:symbol-or-lines>
- **blast_radius**: <list of affected files>
- **evidence**: <1–3 sentences citing specific analyses>
- **reproduction**: <concrete input/sequence or "not visible — requires runtime trace">

### 2. <title>
...

## What I checked but did not find
- <bug class 1>: <one line — why not>
- <bug class 2>: <one line — why not>
- ...
```

## Rules

- Findings ordered by severity desc, then by blast_radius desc.
- If zero findings, still write the "what I checked but did not find" section — that's the proof you did the work.
- Do not read source code directly. Your input is analyses only. This is by design: it forces you to rely on the structured contracts and keeps token cost low.
- If an analysis is missing `assumes` / `enforces` entirely, note it in the "what I checked" section as an audit gap — but do not fabricate findings to fill space.
